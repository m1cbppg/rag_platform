# M10.2 顺序依赖多跳检索最终验收结论

## 验收范围

M10.2 的目标不是让所有问题都执行两跳检索，而是只对“第二跳查询必须依赖第一跳事实”的问题启用顺序链路：

1. 第一跳检索并定位支持证据。
2. 从第一跳证据中抽取受约束的中间事实。
3. 把中间事实注入第二跳查询。
4. 检索第二跳证据，并与第一跳证据合并进入后续精排和回答链路。
5. 任一阶段不满足约束时回退原有检索流程。

## 专项成对 A/B 结果

- 专项数据集：`evaluation/datasets/rag_dependent_multi_hop_v1.jsonl`
- Case 数：15
- 控制变量：两组使用相同 Query Analysis、M9 自适应检索和召回参数，仅切换 DEPENDENT 两跳能力。

| 指标 | 控制组 | 顺序多跳组 | 变化 |
| --- | ---: | ---: | ---: |
| 事实覆盖率 | 83.33% | 100.00% | +16.67pp |
| Recall@5 | 73.89% | 88.33% | +14.44pp |
| Recall@10 | 80.56% | 100.00% | +19.44pp |
| nDCG@10 | 79.86% | 92.34% | +12.47pp |
| 端到端链路成功率 | - | 100.00% | - |
| 回退率 | - | 0.00% | - |

链路内部检查项均为 100%：DEPENDENT 触发、第一跳 Gold 命中、中间事实抽取、支持 Chunk 定位、中间事实准确、第二跳事实注入和第二跳 Gold 命中。

代价是平均延迟从 4548.4 ms 增加到 8769.4 ms，增加 4221.0 ms。因此该能力必须通过 Query 分解门控按需触发，不能用于所有问题。

- Run Code：`M10_2_FINAL_E2E_V2_20260612`
- 执行结果：180/180 完成，失败和运行错误均为 0。
- 顺序依赖链路实际触发：0/180。

这两次全量运行不能作为 M10.2 的严格 A/B，原因包括：

- M10.1 使用动作决策 Prompt `v7-composable-evidence`，M10.2 使用 `v8-composable-evidence-boundary`。
- M10.1 使用查询分解 Prompt `v2`，M10.2 使用 `v3-dependent-two-hop`。
- M10.1 的运行 `top_k=10`，M10.2 的运行 `top_k=20`。
- 两次运行都包含非确定性的 Query 分解、答案生成和 Judge 调用。
- 旧 180 条数据中没有任何 Case 实际进入顺序依赖执行分支。

M10.2 全量运行中的主要额外损失是错误拒答从 8 条增加到 16 条，这与动作决策 Prompt 版本变化直接相关，不能归因于未被触发的顺序依赖执行链路。

## 最终结论

1. **专项能力验收通过**：在真实顺序依赖问题上，M10.2 将事实覆盖率从 83.33% 提升到 100.00%，15 条链路全部执行成功。
2. **工程兼容性验收通过**：原有 180 条评测全部运行完成，未发生异常；其中 0 条触发 DEPENDENT，说明门控没有把旧数据误路由到顺序两跳。
3. **不能宣称整体指标提升**：旧 180 集不包含真实顺序依赖样本，而且对照运行配置不一致。
4. **发布方式**：保留 `dependent_multi_hop_enabled` 配置开关和失败回退；只有 Query 分解明确输出 DEPENDENT 且第一跳事实通过置信度与证据校验时才执行第二跳。
5. **后续优化重点**：动作决策 `v8` 的错误拒答边界需要单独校准，它属于 M8 动作决策问题，不属于 M10.2 顺序依赖检索问题。
