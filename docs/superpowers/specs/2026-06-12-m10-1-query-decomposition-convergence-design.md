# M10.1 查询分解收敛优化设计

## 背景

M10 已完成复杂问题分解、子问题独立检索、RRF 融合、证据配额和结构化
Context。45 条复杂题专项评测表明，查询分解能够提升复杂题的答案和引用质量：

- Judge 通过率从 46.67% 提升到 53.33%；
- Citation Recall 从 56.67% 提升到 64.44%；
- Faithfulness 从 68.33% 提升到 75.78%。

但 180 条全量评测暴露出三个收敛问题：

1. 单独出现“如果”也会进入分解模型候选，导致简单条件题产生无收益调用；
2. 同一规则下的枚举、单流程多步骤、版本比较和二选一决策可能被误拆；
3. 复杂题虽然计划保留 `rerank_top_n + 子问题数` 个结果，但 Qwen Rerank
   实际仍只返回基础 Top5，后续配额恢复无法找回已经被 Rerank 截断的必要证据。

## 目标

M10.1 不增加新功能模块，只让 M10 查询分解能力稳定收敛：

1. 降低简单题、版本比较题和单决策题的误触发；
2. 只接受高置信度、确实需要多份独立证据的分解结果；
3. 复杂题在同一次 Rerank 调用中返回更多候选，避免必要事实被 Top5 提前截断；
4. 保持 M9 原始查询 Anchor、最多两轮检索和异常回退机制不变；
5. 修正动作模型对组合证据和特殊例外的过度拒答；
6. 不新增数据库表。

## 根因分析

### 1. 候选规则过宽

当前候选规则包含单独的“如果”“哪些情况”“什么情况下”。这些词只能证明问题
存在条件，不代表问题包含两个独立信息需求。

例如：

```text
如果修改地址涉及跨省或者有风险，需要什么流程？
```

这是一个带条件的单一流程问题，不需要拆成两个检索任务。

### 2. 模型缺少“分解收益”约束

当前 Prompt 只要求判断是否存在多个信息需求，没有明确排除以下情况：

- 同一个规则中查询多个对象的相同属性；
- 同一个 SOP 中连续执行的多个步骤；
- 新旧版本比较或冲突优先级判断；
- “A 还是 B”“先 A 还是先 B”形式的单一决策。

这些问题拆分后通常不会增加证据覆盖，却会增加一次 LLM 调用和多次检索。

### 3. 扩展 TopN 没有传入 Reranker

Graph 在配额恢复时使用了扩展后的 TopN，但 `RerankService` 和
`Qwen3Reranker` 仍固定使用 `settings.rerank_top_n`。因此必要证据如果排在
第 6 或第 7 位，Graph 根本拿不到该候选，无法进行配额恢复。

## 设计

### 1. 两级保守门控

第一级为确定性门控：

- 问题长度必须达到配置阈值；
- 两个以上问号可以进入模型确认；
- 或命中“同时、并且、以及、分别、另外、然后、在此之前”等强组合信号；
- 单独出现“如果”“哪些情况”“什么情况下”不再触发；
- Query Analysis 已要求澄清时直接跳过；
- 新旧版本、版本冲突和明确二选一决策直接跳过，继续使用 M9 原始链路。

第二级为模型门控：

- 模型必须返回 `benefit_score`；
- 只有 `requires_decomposition=true` 且分数达到配置阈值才采用；
- 要求子问题分别需要独立证据；
- 明确排除同规则枚举、单流程步骤、版本比较和二选一决策；
- 低置信度结果按“无需分解”处理，不视为系统异常。

### 2. 分解类型

模型返回：

```json
{
  "requires_decomposition": true,
  "decomposition_type": "PARALLEL",
  "benefit_score": 0.92,
  "reason": "需要查询两个不同业务规则",
  "sub_queries": []
}
```

M10.1 仅支持 `PARALLEL`。依赖前一步答案才能生成下一步查询的 `DEPENDENT`
问题暂不执行分解，回退 M9 链路。真正的顺序多跳检索单独进入后续模块，避免在
本次收敛中引入新的检索编排风险。

### 3. 动态 Rerank TopN

`RerankService.rerank_documents` 增加可选 `top_n` 参数，并继续兼容所有
现有调用方。

复杂查询使用：

```text
effective_top_n = rerank_top_n + min(子问题数量, 最大额外数量)
```

该值透传到 `Qwen3Reranker` 和百炼接口。同一次请求返回更多结果，不新增一次
Rerank 调用。简单查询继续使用原 Top5。

配额恢复只在扩展后的结果内工作：

- 保持 Qwen Rerank 原始顺序；
- 某个子问题完全没有结果时才追加或替换；
- 不因配额重排已经覆盖全部子问题的结果。

### 4. 配置

新增配置：

```text
query_decomposition_min_benefit_score=0.80
query_decomposition_allow_dependent=false
query_decomposition_rerank_extra_limit=3
```

评测运行快照必须保存这些配置和新的 Prompt 版本。

### 5. 组合证据可回答性

复杂题评测发现，Context 已分别包含两个子问题的直接证据时，动作模型仍可能
因为“没有单个文档逐字描述完整组合场景”而拒答。M10.1 调整可回答性规则：

- 问题中的多个限定条件可以由不同 Context 证据块联合覆盖；
- 同一对象的明确例外规则优先于一般规则；
- 额外状态没有与例外规则冲突时，不要求规则重复枚举该状态；
- 只有条件效果确实未知、证据互相冲突或关键条件完全缺失时才拒答。
- 不能用相邻业务流程代替用户明确询问的流程；
- 用户询问具体标准、完整步骤或明确规则时，相关概述不等于直接答案。

该调整只修改动作判定 Prompt，不绕过精确时间、金额等确定性证据约束。

## 失败处理

- 分解模型缺少新字段或返回非法值：按现有重试机制处理，最终回退原问题；
- 模型返回低收益：正常跳过分解；
- 动态 TopN 未传入：RerankService 使用原配置，保持向后兼容；
- Rerank 外部调用失败：继续使用现有 fail-open；
- 任何新增判断都不能影响普通 M9 检索流程。

## 测试

新增或修改测试覆盖：

1. 单独“如果”不调用分解模型；
2. 多问句仍可进入模型确认；
3. Query Analysis 要求澄清时跳过；
4. 版本比较和二选一决策跳过；
5. 低 `benefit_score` 不采用分解；
6. `DEPENDENT` 默认不采用；
7. 复杂查询向 RerankService 传入扩展 TopN；
8. 简单查询不传扩展 TopN；
9. RerankService、Qwen3Reranker 和失败回退都使用调用级 TopN。

## 验收标准

专项 45 条复杂题：

- Judge 不低于 M10 的 53.33%；
- Fact Coverage 不低于 M10 的 83.33%；
- `CASE_MULTI_HOP_018` 不再因 Rerank Top5 截断丢失必要事实；
- 分解触发数允许下降，但有效改善 Case 不能丢失。

全量 180 条：

- Judge 不低于 66.67%；
- Action Accuracy 不低于 80.00%；
- Fact Coverage 不低于 M9 的 87.82%；
- DIRECT 和 EXACT Fact Coverage 不低于 M9；
- P50 延迟低于 M10 的 12.90 秒；
- REFUSE 至少保持 17/18。
