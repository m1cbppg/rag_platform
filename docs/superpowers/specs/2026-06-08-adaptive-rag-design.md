# 评测驱动的 Adaptive RAG 设计方案

## 目标

把当前已经具备完整功能链路的 RAG 平台，升级为一套评测驱动的
Adaptive RAG 系统，使项目能够体现 3-5 年 AI 应用开发岗位所要求的
技术决策、效果验证、问题诊断和质量控制能力。

目标系统必须能够用可量化证据回答以下三个问题：

1. 不同问题类型分别适合哪种检索策略和切分策略？
2. 系统应当在什么情况下执行改写、拆解、澄清、重试或拒答？
3. 最终答案中的每个重要结论，是否都能被召回证据支持？

## 建设范围

项目使用一套人工约束、模型生成的电商售后实验语料：

- 40 篇源文档
- 300 道评测题
- DeepSeek 负责生成源文档和评测题候选
- Qwen 负责独立审核源文档、标注和系统答案
- 确定性 Python 程序负责计算检索、引用、延迟及费用指标

### 文档数量分布

| 文档类型 | 数量 |
|---|---:|
| FAQ | 12 |
| SOP | 10 |
| RULE | 12 |
| MANUAL | 6 |

### 评测题数量分布

| 题目类型 | 数量 |
|---|---:|
| 单文档直接问答 | 90 |
| 同义表达或模糊表达 | 45 |
| 精确编号、错误码、按钮或字段 | 30 |
| 多条件规则判断 | 45 |
| 多跳问题 | 30 |
| 规则冲突或版本问题 | 20 |
| 无答案或应当拒答 | 40 |

## 版本边界

### V0：冻结现有基线

V0 直接使用现有链路：

```text
Query分析
-> BM25、Vector或Hybrid路由
-> RRF融合
-> Rerank精排
-> Context关系扩展
-> 答案生成
-> 引用格式校验
```

在改变任何检索行为前，必须先运行并保存 V0 的完整评测结果。

### V1：Adaptive Retrieval

增加经过校准的检索质量评分，并且最多执行一轮纠错检索：

```text
初次检索
-> 计算召回质量
-> GOOD：继续后续流程
-> WEAK：改写Query并重试
-> POOR：放宽过滤条件并重试
-> 使用加权RRF融合多轮结果
```

### V2：多跳检索

把复杂的多条件问题拆解成多个原子子问题，每个子问题独立检索。
融合结果时保留每个结果预期支持的 `fact_key`，并计算必要事实覆盖率。

### V3：Claim 级证据校验

把生成答案拆分为原子 Claim。由 Qwen 判断每个 Claim 是否得到所引用
Chunk 的支持。存在无证据或冲突 Claim 时，只允许执行一次答案修正；
修正后仍不满足要求则拒答。

## 评测独立性

同一个模型不能同时生成和评审同一份内容：

- DeepSeek：生成源文档、评测题和被测系统答案
- Qwen：审核文档、审核评测题、执行 LLM-as-a-Judge、校验 Claim 与证据
- Python 程序：计算 Recall、MRR、nDCG、引用指标、延迟和费用

评测系统绝不能把 `reference_answer` 传给被测 RAG。参考答案只能在
系统答案生成完成后，提供给独立 Judge 进行评分。

## 标准证据构建

生成评测题时必须同时生成：

- 标准源文档编码
- 从源文档逐字复制的证据片段
- 事实标识 `fact_key`
- 分级相关度

文档完成导入和 Chunk 切分后，通过映射程序在 `rag_chunk.content`
中定位每一段标准证据。

证据映射状态：

- `MAPPED`：证据能够无歧义地映射到一个或多个 Chunk
- `AMBIGUOUS`：相同证据出现在多个无关 Chunk 中，无法确定唯一来源
- `MISSING`：切分后的 Chunk 中找不到证据，说明解析或切分造成内容丢失

存在 `AMBIGUOUS` 或 `MISSING` 必要证据的题目，不能进入冻结评测集。

## 检索质量模型

V1 的质量判断不能只看召回文档数量，应使用明确的质量特征：

- 候选数量
- BM25 与 Vector Top-K 重合率
- Rerank Top1 分数
- Rerank Top3 平均分
- Rerank Top1 与 Top2 分差
- 不同源文档数量
- 目标文档类型覆盖率
- 可选的 Qwen 可回答性评分

质量阈值只能使用开发集校准，使用验证集选择最终配置。执行测试集前
必须冻结阈值，禁止根据测试集结果继续调参。

## 数据集划分

按照源文档主题和版本组进行划分，不能随机打散全部问题：

- 开发集：60%
- 验证集：20%
- 测试集：20%

同一规则的不同版本以及由其生成的问题必须进入同一数据划分，避免
近重复规则分别出现在训练调试数据和测试数据中造成泄漏。

## 核心指标

### 检索指标

- Recall@1/3/5/10
- MRR
- nDCG@5/10
- 多跳问题必要事实覆盖率

### 答案指标

- Faithfulness：答案是否忠实于上下文
- Answer Relevance：答案与问题的相关性
- Completeness：必要事实是否回答完整
- Citation Entailment：引用是否真正支持对应结论
- 拒答与澄清的 Precision、Recall、F1

### 性能与费用

- 端到端 p50/p95 延迟
- 实际检索轮数
- 输入和输出 Token
- 单题估算费用

## 完成标准

只有满足以下条件，增强项目才算完成：

1. V0-V3 使用完全相同的冻结测试集运行。
2. 每次实验都保存完整配置和代码提交SHA。
3. V1 在延迟增幅可接受的前提下，提高 Recall@5 或 MRR。
4. V2 提高多跳题的必要事实覆盖率。
5. V3 降低无证据 Claim 比例，并提高 Citation Entailment。
6. 所有对外描述的优化都同时给出基线值、优化后绝对值和提升量。
