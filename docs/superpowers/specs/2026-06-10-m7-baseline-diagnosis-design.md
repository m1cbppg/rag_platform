# M7 全量基线评测与错误归因设计

## 1. 目标

M7 的目标不是继续增加新的 RAG 功能，而是使用 M6 已经建立的自动化评测执行器，对冻结数据集的 DEVELOPMENT 分片执行完整基线实验，并回答以下问题：

1. 当前 RAG 的核心指标是多少；
2. 失败主要发生在融合召回、精排、Context 构建、行为决策、答案生成还是引用环节；
3. 哪些题型和难度最差；
4. M8 应优先优化哪个环节；
5. 后续实验相对 V0 基线是否真正提升。

冻结数据集 `rag_eval_ecommerce:v1` 当前包含 40 份文档、300 道评测题，其中 DEVELOPMENT 分片有 180 道题。

## 2. 约束

1. 不修改冻结数据集和 Gold Evidence；
2. 不把参考答案传入被测 RAG；
3. 不修改现有 RAG 的检索、拒答和答案生成行为；
4. 不新增数据库表，避免在尚未稳定的归因规则上固化表结构；
5. 逐题归因结果写入 JSON 报告，汇总结果写入 Markdown 报告；
6. 归因必须区分数据库中可直接证明的事实和基于指标做出的推断。

## 3. 总体架构

M7 分为四个组件：

### 3.1 评测观测增强

扩充 M6 的 `ChatServiceEvaluationAdapter`：

- 融合结果保存完整 metadata；
- Hybrid 模式保存 RRF 融合分数；
- 保留 `sources`、`bm25_rank`、`vector_rank`、各通道原始分数；
- 继续保存 RERANK 和 FINAL 两个阶段的 Chunk 排名。

这些数据用于判断 Gold Chunk 在融合、精排和最终 Context 中分别出现在哪里。

### 3.2 诊断数据读取

在 `DatasetRepository` 增加只读诊断查询：

- 运行及逐题结果；
- 问题、参考答案、题型、难度；
- Judge 分数和理由；
- 每个阶段的 Retrieval Hit；
- 每道题的 Gold Chunk、Fact Key 和证据文本。

读取层只负责还原事实，不执行归因。

### 3.3 规则归因引擎

归因引擎是纯 Python 逻辑，不依赖数据库和模型。对每道题依次检查：

1. 运行是否发生异常；
2. 预期行为和实际行为是否一致；
3. 必要事实是否进入融合召回；
4. 融合召回中的必要事实是否被精排保留；
5. 精排结果中的必要事实是否进入最终 Context；
6. 系统是否错误拒答或错误回答；
7. 答案完整性、忠实度、相关性、冲突处理和引用是否通过。

主要归因包括：

- 执行异常；
- 召回完全缺失；
- 召回部分缺失；
- 精排完全淘汰；
- 精排部分淘汰；
- Context 完全丢失；
- Context 部分丢失；
- 错误拒答；
- 错误回答；
- 缺少澄清能力；
- 答案不完整；
- 答案不忠实；
- 答案不相关；
- 引用失败；
- 冲突处理失败；
- Judge 缺失；
- 通过。

每道题包含一个主因和零到多个次因。报告按主因统计，次因用于保留更完整的诊断信息。

## 4. 归因优先级

归因按照上游优先原则：

```text
执行异常
→ 行为契约问题
→ 融合召回
→ 精排
→ Context
→ 错误拒答
→ 引用
→ 答案质量
```

例如，必要事实根本没有进入融合召回时，即使最终答案完整性为 0，主因仍然是“召回缺失”，而不是“答案不完整”。这样可以避免把上游缺陷重复统计为多个下游问题。

对于多事实题，分别计算融合、精排和最终 Context 的 Fact Coverage：

- 0：完全缺失；
- 大于 0 且小于 1：部分缺失；
- 1：全部覆盖。

## 5. 报告结构

M7 输出两个文件：

1. `evaluation/reports/<run_code>.json`
   - 完整运行配置；
   - 汇总指标；
   - 每道题的阶段覆盖率；
   - 主因、次因、证据和建议。

2. `evaluation/reports/<run_code>.md`
   - 中文执行摘要；
   - 核心指标；
   - 题型和难度切片；
   - 失败归因分布；
   - 检索漏斗；
   - 典型失败案例；
   - M8 优化优先级。

## 6. 基线运行

使用以下固定配置运行 DEVELOPMENT 分片：

- 数据集：`rag_eval_ecommerce:v1`
- 实验版本：`V0`
- 实验名称：`baseline-hybrid-rrf-rerank`
- Run Code：`V0_DEVELOPMENT_20260610`
- Top K：20
- 并发：3
- Judge：百炼 Qwen

相同 Run Code 可以断点续跑。运行成功表示评测流程完成，不代表答案质量通过。

## 7. 完成标准

1. 180 道 DEVELOPMENT 题全部产生成功或失败记录；
2. 每条成功执行的题都有 Judge 结果；
3. 每道题都能生成一个明确的主归因；
4. 中文 Markdown 和 JSON 报告均可从数据库重复生成；
5. 全量测试、MySQL 集成测试和格式检查通过；
6. 报告能够给出有数据依据的 M8 优化优先级。
